# 统一语言（Ubiquitous Language）

## 1. 使用规则

本文术语在产品文档、代码、API、事件、数据库和 UI 中保持一致。禁止用技术层名词替代业务概念，例如把所有对象称为 `record`、`entity`、`data`；也禁止同一概念在不同模块使用多个含义。

术语的英文名称用于类、事件和契约；中文名称用于产品界面和说明。必要的历史快照使用 `Snapshot` 后缀，避免与当前主数据混淆。

## 2. 核心交易术语

| 中文 | 英文 | 精确定义 | 不等同于 |
|---|---|---|---|
| 商业客户 | Partner | 与租户建立 B2B 交易关系的组织，具有状态、资格、条款和联系人 | 登录用户、供应商、消费者 |
| 客户采购人 | Customer Buyer | 代表商业客户查看和接受报价的外部用户 | 内部销售 |
| 酒款 | Wine Product | 与生产者、产区、分类相关的抽象产品，不含年份和包装维度 | 可售 SKU |
| SKU | Stock Keeping Unit | 由酒款、年份/NV、容量、包装数量和包装形态确定的交易单元 | 库存批次 |
| 供给池 | Supply Pool | 具有位置、贸易属性和可用承诺方式的供给来源集合 | 物理仓库本身 |
| 库存批次 | Inventory Lot | 某 SKU 在某供给池中的可数量化批次，具有在手、预占和批次属性 | 报价行或订单行 |
| 可用量 | Available Quantity | `on_hand - reserved` 的即时计算或受控投影 | 客户承诺、未来供给 |
| 可承诺供给 | Promisable Supply | 经供给类型、时效和规则判断可用于报价评估的供给信息 | 已预占库存 |
| 询价 | Inquiry | 客户需求的非约束性输入，可选 P2 对象 | 报价 |
| 报价 | Quotation | 针对客户、商品、数量、价格、路径、有效期和条款形成的有版本商业提议 | 订单、价格表 |
| 报价修订 | Quotation Revision | 同一报价业务编号下不可覆盖的修订版本 | 原地修改已发送报价 |
| 报价行 | Quotation Line | 报价中的 SKU、数量、价格和供给快照 | SKU 主数据 |
| 商业快照 | Commercial Snapshot | 在决定时冻结的客户、商品、价格、条款和路径事实 | 对主数据的实时引用 |
| 客户接受 | Quotation Acceptance | 客户对特定报价修订的幂等终态决定 | 订单创建本身 |
| 贸易订单 | Trade Order | 由已接受报价唯一转换、用于库存和履约执行的商业承诺 | 购物车、采购单 |
| 预占 | Reservation | 为订单将库存从可用状态转为已承诺但未消费 | 出库、锁表 |
| 消费预占 | Reservation Consumption | 出库或履约确认后把已预占数量转为已消耗 | 释放预占 |
| 释放预占 | Reservation Release | 取消或失败后把未消费数量恢复为可用 | 增加在手库存 |

## 3. 贸易规划术语

| 中文 | 英文 | 定义 |
|---|---|---|
| 贸易路径 | Trade Route | 一种抽象交付方案，描述可用供给、目的地、时效、费用和履约模板，不代表法律建议 |
| 路径资格 | Route Eligibility | 客户是否允许使用某贸易路径的生效配置 |
| 硬约束 | Hard Constraint | 不满足即不能选择路径的规则 |
| 拒绝原因 | Rejection Reason | 对硬约束失败的稳定机器码、参数和安全解释 |
| 路径评分 | Route Score | 对有效路径按版本化策略计算的 0~100 分数 |
| 推荐路径 | Recommended Route | 当前评估中得分最高且满足所有硬约束的路径 |
| 路径覆盖 | Route Override | 授权人员选择非推荐但有效路径的决定，必须记录原因 |
| 策略版本 | Policy Version | 对硬约束、权重和阈值的一组不可变配置标识 |
| 供给置信度 | Supply Confidence | 对供给可在目标时间兑现的演示评分，不是统计预测 |
| 操作复杂度 | Operational Complexity | 对步骤数量、人工依赖和外部依赖的相对评分 |

路径代码：

- `SH_GENERAL_TRADE`：上海一般贸易交付的抽象演示路径；
- `NB_BONDED_B2B`：宁波保税 B2B 交付的抽象演示路径；
- `HK_FREE_TRADE`：香港自由贸易交付的抽象演示路径。

不得将这些代码解释为真实报关、税务或监管结论。

## 4. 供给类型

| 代码 | 中文 | P1 行为 |
|---|---|---|
| `DOMESTIC_ON_HAND` | 国内现货 | 可自动预占 |
| `BONDED_ON_HAND` | 保税现货 | 可自动预占，受路径资格限制 |
| `HONG_KONG_ON_HAND` | 香港现货 | 可自动预占，受路径资格限制 |
| `IN_TRANSIT_PRESALE` | 在途预售 | 可用于报价参考，需人工确认，不自动预占 |
| `OVERSEAS_SOURCING` | 海外代采 | 仅形成预计供给方案，需人工确认 |

## 5. 报价和审批术语

| 中文 | 英文 | 定义 |
|---|---|---|
| 列表价 | List Price | 定价输入中的基准价格，不保证等于公开市场价 |
| 净单价 | Net Unit Price | 应用折扣/调整后客户支付的单位价格 |
| 预计成本 | Estimated Cost | 仅授权内部角色可见的演示成本快照 |
| 预计毛利 | Estimated Margin | 基于演示成本计算的评估值，不是财务总账结果 |
| 审批规则 | Approval Rule | 决定报价是否需要人工批准的版本化规则 |
| 审批项 | Approval Task | 对一个具体报价修订的待决工作项 |
| 退回修改 | Changes Requested | 审批者不作最终拒绝，要求销售创建修订 |
| 发送 | Send/Issue | 使已批准报价可被指定客户查看 |
| 撤销 | Withdraw | 在客户接受前使报价不可继续接受 |
| 过期 | Expire | 有效期到达后不能接受的终态 |

## 6. 履约术语

| 中文 | 英文 | 定义 |
|---|---|---|
| 履约模板 | Fulfillment Template | 某路径的版本化步骤、依赖、SLA 和可见性定义 |
| 履约计划 | Fulfillment Plan | 为一个订单从模板实例化的执行计划 |
| 履约步骤 | Fulfillment Step | 可分配、开始、完成、失败或逾期的最小工作单位 |
| 前置依赖 | Dependency | 必须完成后才能开始当前步骤的关系 |
| 里程碑 | Milestone | 对业务有意义的履约事实，可标记为公开或内部 |
| SLA | Service-Level Target | 演示用目标完成时间，不是对真实客户的法律承诺 |
| 公开时间线 | Public Timeline | 客户被授权查看的里程碑投影 |
| 内部时间线 | Internal Timeline | 包含工作、异常和技术关联信息的授权投影 |

## 7. 异常与恢复术语

| 中文 | 英文 | 定义 |
|---|---|---|
| 异常 | Exception Case | 需要明确责任和恢复动作的业务/技术偏差对象 |
| 检测事实 | Detection Fact | 创建异常的不可变触发信息 |
| 负责人 | Assignee | 当前被授权推进异常的人 |
| 恢复动作 | Recovery Action | 受控、可审计、可重试的修复命令 |
| 重试 | Retry | 对相同业务意图再次执行，必须幂等 |
| 重放 | Replay | 对已持久化事件再次交付，不等同于重新发起业务命令 |
| 死信/失败发布 | Failed Publication | 达到自动重试阈值后需要系统操作员处理的技术状态 |
| 补偿 | Compensation | 用新的业务事实抵消或恢复此前事实，不删除历史 |

## 8. 应收术语

| 中文 | 英文 | 定义 |
|---|---|---|
| 应收 | Receivable | 与订单相关的应收金额、到期和余额对象 |
| 付款记录 | Payment Record | 财务人员登记的外部付款事实，不代表真实网关交易 |
| 部分付款 | Partial Payment | 付款金额小于应收剩余金额 |
| 冲正 | Reversal | 以关联负向记录抵消错误付款 |
| 逾期 | Overdue | 到期时间已过且余额大于零 |
| 外部参考号 | External Reference | 财务录入的幂等业务键，不得包含秘密信息 |

## 9. 技术协作术语

| 术语 | 定义 |
|---|---|
| Domain Event | 聚合内已经发生的事实，服务于本应用协作 |
| Integration Event | 对其他模块/外部消费者发布的稳定版本化事件 |
| Reliable Publication | 业务事务提交后不会因进程崩溃永久丢失的事件发布机制 |
| Outbox | 对外 Kafka 事件的持久化发布记录；不要求所有本地事件都手写 Outbox 表 |
| Inbox | 消费者按 event ID 去重和保存处理结果的记录 |
| Correlation ID | 关联一次业务旅程的标识 |
| Causation ID | 表示当前动作由哪个命令/事件引起 |
| Idempotency Key | 标识同一业务意图的客户端/业务键 |
| Read Model | 为查询和页面优化的投影，不承载源聚合不变量 |
| Tenant | 数据和权限隔离的商业组织空间 |

## 10. 避免使用的模糊词

- “库存锁定”：必须说明是数据库锁、预占业务事实还是 UI 展示；
- “支付”：P1 只登记付款事实，不连接支付网关；
- “实时”：必须给出一致性和延迟范围；
- “自动选择”：必须说明硬约束、评分、版本和覆盖机制；
- “全链路”：必须说明覆盖到应收登记，不声称真实海关/WMS/总账；
- “智能”：没有可解释算法和验证时不使用；
- “微服务”：当前架构是模块化单体。
