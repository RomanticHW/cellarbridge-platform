# 错误、补偿与恢复

## 1. 错误分类

| 类别 | 示例 | HTTP/处理 | 是否自动重试 |
|---|---|---|---|
| 输入错误 | 数量 <= 0、格式非法 | 400 | 否 |
| 权限错误 | 无审批权限 | 403 | 否 |
| 业务冲突 | 报价已过期、状态非法 | 409 | 否，用户刷新/改动作 |
| 并发冲突 | version 不匹配 | 412/409 | UI 重新加载，不盲重试 |
| 资源不足 | 库存不足 | 409 | 需人工或供给变化后重试 |
| 瞬时技术错误 | DB 连接短暂失败 | 503 | 有界重试 |
| 异步处理失败 | 消费者异常 | 内部失败状态 | 指数退避、死信/异常 |
| 契约错误 | 事件缺字段/hash 不一致 | 严重告警 | 不自动无限重试 |

## 2. 补偿原则

- 不使用跨模块 XA/2PC；
- 每个模块提交本地事实；
- 后续失败通过新的命令和事实补偿；
- 已对客户生效的商业事实不静默删除；
- 补偿必须可重试、幂等、有权限和审计；
- 无安全补偿时进入人工异常，不伪造成功。

## 3. 场景恢复矩阵

| 失败点 | 已提交事实 | 恢复方式 | 不允许 |
|---|---|---|---|
| 报价接受后订单创建失败 | Quotation ACCEPTED | 重放接受事件/显式创建命令，quote ID 幂等 | 撤销接受事实掩盖技术失败 |
| 订单创建后预占失败 | Order + failed result | 补充供给后批准重试，或取消订单 | 保留部分预占 |
| 预占成功后计划创建失败 | Reservation CONFIRMED | 重放事件/创建计划幂等 | 再次重复预占 |
| 履约步骤适配器超时 | Step IN_PROGRESS/attempt | 查询外部幂等状态或受控重试 | 直接标完成 |
| 取消时释放失败 | Order CANCELLATION_PENDING | 异常中心重试释放 | 订单直接 CANCELLED |
| 付款登记重复 | Payment 已存在 | 外部参考号返回已有 | 第二次增加余额 |
| 报表投影失败 | 源事件已提交 | 重放/重建投影 | 修改源业务数据 |

## 4. 重试策略

- 只对可判定瞬时且幂等的操作自动重试；
- 使用指数退避、抖动、最大次数；
- 重试上下文保存 attempt、lastErrorCode、nextAttemptAt；
- 业务不足/权限/状态错误不自动重试；
- 外部适配器必须使用幂等键或先查询状态；
- 达到阈值创建系统操作异常和指标；
- 手动重试仍通过相同幂等机制。

## 5. 事件发布恢复

可靠发布记录状态：`PENDING`, `PUBLISHED`, `FAILED_RETRYABLE`, `FAILED_FINAL`。

- 业务事务与 PENDING 记录同事务；
- 发布成功后记录 broker 元数据；
- 崩溃窗口允许重复，不允许永久丢失；
- 消费者 Inbox 消除重复副作用；
- FAILED_FINAL 只能由系统操作员在查看错误/契约后重放；
- 重放写审计，不能编辑原 payload；需要修正时发布补偿/新版本事件。

## 6. 对账任务

周期性 reconciliation 检查：

- ACCEPTED 报价长时间无订单；
- PENDING_RESERVATION 订单无处理结果；
- CONFIRMED 预占无计划；
- CANCELLATION_PENDING 超时；
- OPEN 应收余额与付款净额不一致；
- 投影 checkpoint 长时间不前进。

对账只创建异常/补发幂等命令，不直接跨模块修表。

## 7. 用户可恢复体验

错误响应包含：

- 稳定 code；
- 简洁 title/detail；
- 当前状态/版本（安全时）；
- `retryable`；
- 建议动作；
- traceId；
- 字段错误列表。

UI 对不同错误给出“刷新、修改输入、申请审批、稍后重试、联系系统操作员并提供 traceId”等具体操作，不只显示“系统错误”。

## 8. 演练

发布前至少演练：

1. 订单事务提交后进程终止；
2. Kafka 不可用 5 分钟后恢复；
3. 同一事件重复 100 次；
4. 库存热点并发和死锁重试；
5. 履约适配器超时但外部实际成功的模糊结果；
6. 报表投影清空后重建；
7. 错误付款冲正；
8. 双租户越权探测。
