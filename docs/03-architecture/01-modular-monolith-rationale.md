# 为什么选择模块化单体

## 1. 决策摘要

CellarBridge 以一个可部署 Java 应用承载多个严格隔离的业务模块。当前不选择微服务，也不采用无边界传统单体。

## 2. 业务与团队假设

- 演示项目由一名主要开发者维护；
- 业务链路有较多本地一致性、不变量和跨模块学习成本；
- 审阅者需要在笔记本上快速运行；
- 无独立团队、独立发布节奏或明确差异化伸缩需求；
- 目标是证明边界和工程判断，而非部署单元数量。

在这些假设下，微服务会增加网络失败、契约部署、环境、追踪、数据一致性和本地运行成本，却没有相应组织收益。

## 3. 与传统分层单体的区别

传统分层：

```text
controllers/
services/
repositories/
entities/
```

这种结构容易让 OrderService 直接调用 InventoryRepository、CatalogEntity 在全局共享、事务跨越任意表。

CellarBridge：

```text
modules/
  quotation/
  tradeorder/
  inventory/
  fulfillment/
```

每个模块：

- 拥有语言、聚合、用例和表；
- 对外只暴露少量 API/events；
- internal 包不可被引用；
- 通过架构测试执行边界；
- 可以独立进行模块集成测试和文档生成。

## 4. 与微服务的权衡

| 维度 | 模块化单体 | 微服务 | 当前判断 |
|---|---|---|---|
| 本地运行 | 一应用，容易 | 多进程/网络/基础设施 | 单体胜 |
| 本地事务 | 直接明确 | saga/最终一致 | 核心库存需要本地事务 |
| 边界 | 需自动验证 | 网络天然但仍可逻辑耦合 | Modulith + ArchUnit 足够 |
| 独立发布 | 不支持模块独立部署 | 支持 | 当前无需求 |
| 独立伸缩 | 进程级 | 服务级 | 当前无证据 |
| 故障隔离 | 较弱 | 较强但更复杂 | 演示 profile 不需要 |
| 运维成本 | 低 | 高 | 单体胜 |
| 审阅清晰度 | 高（若边界执行） | 容易被基础设施淹没 | 单体胜 |

## 5. 如何防止退化

- package-by-domain；
- `internal` 封装和 named interfaces；
- 模块 schema 所有权；
- 禁止跨模块 FK/Repository；
- 发布事件和显式查询 API；
- 每模块独立迁移目录；
- 模块依赖图进入 CI；
- ADR 审核例外；
- `shared` 仅允许极少技术原语（Clock、TenantId、Money contract 等需审查）。

## 6. 未来提取信号

只有出现可量化证据才考虑提取：

- 某模块有独立团队和发布节奏；
- 负载/资源模型显著不同；
- 故障隔离的商业价值超过复杂度；
- 公开契约稳定且不依赖同事务；
- 数据迁移和运维能力就绪；
- 本地开发体验有可接受替代。

候选通常可能是 reporting、notification 或外部 integration，而不是为了简历先拆 Order/Inventory。

## 7. 非决定

此决策不意味着：

- 禁止 Kafka；外部事件 full profile 仍可使用；
- 所有模块必须同步调用；长流程仍采用可靠事件；
- 所有表可互相 join；模块数据边界仍严格；
- 不需要契约；OpenAPI、AsyncAPI 和模块接口仍版本化；
- 未来绝不拆分；只是当前没有足够理由。
