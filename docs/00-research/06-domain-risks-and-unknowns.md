# 领域风险与未知项

## 1. 目的

该文档防止产品在证据不足时假装“懂业务”。未知项被分成阻塞、可配置和明确排除三类。只有阻塞项需要在实现前获得决策；其余通过配置、端口或范围控制处理。

| ID | 未知项 | 风险 | 处理方式 |

## 3. 技术风险

### 3.1 过度拆分

风险：为展示微服务而复制网关、配置中心、注册中心和分布式事务，增加运行门槛却没有独立部署价值。

控制：模块化单体；通过模块边界、事件和 Schema 所有权体现可拆分性；提取服务必须满足 ADR-001 中的触发条件。

### 3.2 事件驱动掩盖业务失败

风险：事件异步后，订单在失败时长时间停留在中间状态，且评审者不知道如何恢复。

控制：所有长流程状态可见；持久化事件发布；有限重试；异常中心接管；提供重放和人工恢复路径。

### 3.3 缓存导致库存错误

风险：将库存可用量放入 Redis 并按缓存判断，造成超卖或错误拒绝。

控制：库存读缓存只用于提示；最终预占必须在 PostgreSQL 中原子校验；缓存失效不影响正确性。

### 3.4 多租户数据泄漏

风险：分页后过滤、忘记 tenant 条件、错误使用管理员接口或报表跨租户聚合。

控制：租户上下文、Repository 规则、组合唯一约束、权限测试、日志脱敏和可选 PostgreSQL RLS 防御层。

### 3.5 规格与代码漂移

风险：OpenAPI、状态机、README 和实现逐渐不一致。

控制：契约验证、架构测试、需求追踪、同 PR 更新、实现状态表、发布清单。

### 3.6 依赖版本过新

风险：Spring Boot 4.1 和相关生态可能存在兼容变化。

控制：使用官方 BOM、固定版本、最小依赖、兼容性验证；若必须回退，需要 ADR，不允许局部混搭。

## 4. 法律与公开仓库风险

- 不使用研究对象商标作为项目图标或仓库名称；
- 不把招聘截图放入公开仓库；
- 不抓取或提交网站批量内容；
- 不使用真实品牌酒款、酒评和价格；
- 不声称能执行真实海关、税务或支付业务；
- 不在 Git 历史中先提交再“计划删除”私密材料；
- 依赖和素材必须有清晰许可证。

## 5. 阻塞决策清单

当前设计基线没有阻塞项。实现开始前必须确认：

1. GitHub 公共仓库和私有控制仓库的所有者；
2. 本机 Java、Node、Docker 和 GitHub CLI 可用；
3. Spring Boot 4.1 依赖组合通过最小启动验证；
4. 公共仓库只导入 public bundle，私有 bundle 不得混入；
5. 端口和默认账号不与本机服务冲突。
