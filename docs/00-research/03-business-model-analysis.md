# 经营模式分析

## 1. 核心判断

公开资料显示的能力链不是“卖一瓶酒”的单点交易，而是围绕商业客户组织国际货源、商品内容、报价、交付路径和供应链执行。其最大概率经营模式可概括为：

> **通过数字化目录与供应链运营能力连接海外酒款供给和国内商业客户，以贸易成交和履约服务实现收入。**

该模式同时具备平台属性和重运营属性。平台降低信息搜索、匹配和协同成本；运营团队承担采购组织、仓储物流、单证与客户交付等工作。

## 2. 商业模式画布

### 2.1 客户群体

以下为 `INFERRED`，不代表公司公开确认的精确分类：

- 葡萄酒经销商与区域酒商；
- 精品零售门店和连锁零售；
- 餐厅、酒店及会所采购；
- 电商经营者和内容型酒类商家；
- 需要海外代采、在途预售或多路径交付的专业买家。

CellarBridge 将上述群体统一抽象为 `Commercial Partner`，通过客户类型、交付区域、付款条款和渠道资格表达差异，不在核心领域为每个行业复制流程。

### 2.2 价值主张

- 汇集分散的海外酒款、年份、包装与供给信息；
- 降低商业客户寻找货源和询价的沟通成本；
- 将价格、成本、交付时间和路径变成可比较方案；
- 提供从成交到履约的责任和状态可见性；
- 通过批次、节点和审计记录提高可追溯性；
- 通过多种贸易交付路径覆盖不同客户条件。

### 2.3 关键活动

- 商品与供应商资料维护；
- 海外采购和供给组织；
- 客户询价、报价和商务审批；
- 交付路径与成本构成评估；
- 订单、库存和仓库协同；
- 国际/国内物流节点管理；
- 异常跟进和客户沟通；
- 对账、应收和经营分析。

### 2.4 关键资源

- 酒款、酒庄、年份、价格和供给数据；
- 海外采购网络；
- 上海、宁波、香港相关交付能力；
- 供应链运营团队；
- 客户关系和历史交易数据；
- 自建供应链业务程序。

### 2.5 关键合作方

以下属于场景推断：

- 海外酒庄、酒商或供货商；
- 国际货运、集运和国内承运商；
- 仓库与保税仓运营方；
- 报关、单证和检验相关服务方；
- 银行、支付和开票系统；
- 企业客户采购与财务团队。

项目不模拟这些机构的真实接口，只定义 `Supplier`, `Warehouse`, `Carrier`, `Compliance Provider`, `Payment Provider` 等外部端口。

### 2.6 收入来源假设

| 收入来源 | 置信度 | CellarBridge 是否建模 |
|---|---|---|
| 商品进销差价 | 中高 | 通过成本快照、报价和毛利规则建模 |
| 海外代采服务费 | 中 | 作为价格构成项，不单独结算 |
| 物流与履约服务费 | 中 | 作为路径价格构成项 |
| 特定交付路径服务费 | 中 | 作为路径价格构成项 |
| 软件订阅费 | 低 | 不建模 |
| 广告或内容推广费 | 低 | 不建模 |

### 2.7 成本结构假设

- 商品采购成本；
- 国际与国内物流成本；
- 仓储、操作和单证成本；
- 资金占用、汇率和坏账风险；
- 人工运营与客户服务；
- 系统研发和数据维护。

## 3. 经营模式中的关键矛盾

### 3.1 丰富供给与准确承诺

货源可以来自国内现货、保税库存、在途货物和海外代采。展示丰富供给容易，给出可兑现的数量、价格和交期更难。产品必须区分“可搜索”“可报价”“可预占”“已承诺”。

### 3.2 多路径灵活性与运营复杂度

多路径提升成交机会，但带来不同的客户资格、库存位置、成本项、交付节点和失败方式。若路径只作为订单上的一个枚举，系统无法解释方案，也无法生成正确履约步骤。

### 3.3 快速报价与商业控制

销售希望快速响应客户，管理者需要控制折扣、毛利、账期和特殊承诺。报价流程必须支持草稿效率、自动规则和有证据的人工审批，而不是所有报价都走同一审批链。

### 3.4 业务协同与历史真实性

商品名称、客户地址、成本和政策会变化。订单历史必须保持当时承诺，不能依赖实时主数据回放。快照和审计因此是核心能力，不是报表附属功能。

## 4. 产品指标建议

这些指标用于演示平台的经营价值，不代表研究对象真实 KPI：

- 报价首次响应时间；
- 报价审批周期；
- 报价接受率；
- 路径推荐被人工覆盖比例及原因；
- 订单转换幂等冲突数；
- 库存预占成功率和缺货原因；
- 履约按时完成率；
- 开放异常数量、平均解决时间；
- 应收逾期金额；
- 关键操作审计完整率。

## 5. 对技术架构的直接影响

- 报价、订单、库存、履约是独立聚合和模块；
- 路径评估使用规则版本和可解释结果；
- 主数据与交易数据之间使用快照隔离；
- 长流程采用事件和过程管理器，而非跨模块大事务；
- 热点库存写入使用显式 SQL 和并发测试；
- 报表使用异步读模型，避免跨模块联表；
- 权限不仅区分角色，还限制成本、毛利和租户数据可见性。
