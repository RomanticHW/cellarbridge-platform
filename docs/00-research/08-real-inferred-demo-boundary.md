# 真实事实、合理推断与演示增强边界

## 1. 为什么必须明确边界

技术作品最容易失去可信度的方式，是把公开公司介绍扩写成未经证实的“内部系统”。CellarBridge 不这样做。每项能力都注明其来源属性，并在 README 和演示中保持同一口径。

## 2. `CONFIRMED`：可由来源支持

- 2014 年在成都成立；
- 国际酒饮交易服务定位；
- 大搜酒 WineMatcher；
- 自建供应链业务程序，覆盖商品、物流、清关、客户数字化；
- 波尔多、勃艮第及法国精品酒饮供给；
- 上海一般贸易、宁波跨境 B2B、香港自由贸易三种交付方式；
- 在途预售、海外代采和法国至上海零担物流发展经历；
- 2023 年采购酒款超过 2,500 个、服务超过 2,000 家商业客户的公司自述。

## 3. `INFERRED`：根据事实形成的产品假设

- 主要客户是酒商、零售、餐饮酒店和电商经营者；
- 商业客户价格、账期和服务条件存在差异；
- 收入来自商品毛利与供应链服务；
- 销售、贸易运营、仓库和财务需要协同；
- 业务需要客户资格、批次、在途和交付节点数据；
- 特殊折扣或低毛利报价需要审批。

这些假设可被未来信息修正，不写成不可变事实。

## 4. `DEMO_ENHANCEMENT`：为完整产品和技术评审增加

- 可解释贸易路径评分引擎；
- 版本化价格构成和策略快照；
- 单级销售经理审批；
- OIDC 多角色权限；
- 报价接受后的幂等订单转换；
- 原子库存预占和并发压力测试；
- 事务性事件发布与 Kafka 外部化；
- Inbox 去重、异常中心和 SLA 监控；
- 简化客户信用额度、应收和付款；
- 多租户隔离、审计和经营看板；
- React 客户轻门户。

## 5. 对外表述范例

### 正确

> 本项目依据公开资料中提到的多种交付方式，设计了一个可解释的贸易路径评估模块。该模块是为了展示规则建模和架构能力而增加的产品设计。

### 错误

> WineMatcher 内部使用了我们复刻的路径评分算法。

### 正确

> 公开资料显示公司自述拥有 2,500 多个国际采购酒款。基于该规模，本项目优先使用 PostgreSQL 全文与模糊搜索，而没有引入独立搜索集群。

### 错误

> 公司生产数据库只有 2,500 条商品数据，所以 Elasticsearch 没有必要。

## 6. 实现标记

代码中不需要用注解标记三类来源，但下列位置必须保留分类：

- 研究和证据矩阵；
- 产品范围和 ADR；
- README 非官方声明；
- 演示讲解稿；
- 任何涉及研究对象能力的公开说明。

合成数据不得使用研究对象、真实客户或真实酒款名称，以避免事实和演示混淆。
