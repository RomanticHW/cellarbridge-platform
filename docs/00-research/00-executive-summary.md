# 调研执行摘要

## 1. 研究目标

本研究不是为了复刻某家公司的内部系统，而是为了回答三个工程问题：

1. 公开信息能够支持怎样的经营模式判断；
2. 哪个业务场景最能代表该类公司的真实价值链；
3. 如何将该场景抽象成一个边界清晰、可运行、可测试、适合 Java 技术评审的企业级产品。

研究对象为成都优自西方国际贸易有限公司及其公开提及的 FineWest、大搜酒 WineMatcher、ShunShip 等业务名称。核心证据来自项目发起人提供的 BOSS 直聘公司介绍截图，辅以公司官网存在性、行业通用流程和官方技术资料。由于未获得内部流程、合同、ERP/WMS 接口、财务口径或监管配置，本仓库严格区分事实、推断和演示增强。

## 2. 主要结论

### 2.1 已确认的经营特征

依据来源 `S-001`，可以确认以下事实：

- 公司于 2014 年在成都成立，定位为以数据技术驱动的国际酒饮交易服务公司；
- 对外产品或服务名称包括大搜酒 WineMatcher 和 ShunShip；
- 业务经历了团购与个人零售、B2B 在途预售、海外代采、法国至上海零担国际物流、内容驱动的 B2B 交易、香港交付和宁波保税相关交付等阶段；
- 公司强调自建酒饮供应链业务程序，并对商品、物流、清关和客户进行数据化管理；
- 对外介绍了三类交付路径：上海一般贸易交付、宁波跨境 B2B 交付、香港自由贸易交付；
- 公开材料称 2023 年国际采购酒款超过 2,500 个，并服务超过 2,000 家国内商业客户；
- 酒款供给重点提及波尔多、勃艮第及法国精品酒饮。

### 2.2 最大概率的经营模式

综合上述证据，本研究判断其核心不是单纯的软件订阅，也不是单纯的消费者电商，而是：

> **以数字化商品与供应链能力为支撑，为商业客户提供酒款发现、询报价、采购组织、多路径交付和履约协同的一体化 B2B 交易服务。**

这是一种“平台能力 + 贸易履约服务”的混合模式。可能的收入来源包括商品进销差价、采购服务、物流与履约服务、特定交付路径服务费等；这些收入结构没有公开财务资料直接证明，因此在仓库中标记为 `INFERRED`。

### 2.3 最适合演示的业务场景

候选场景包括消费者商城、酒款内容搜索、海外采购管理、国际物流跟踪、B2B 询报价与履约协同。经业务匹配度、Java 技术表现力、端到端完整性、审阅便利性和实现风险加权后，最高分场景为：

> **商业客户询价 → 客户专属报价 → 多贸易路径评估 → 审批 → 客户接受 → 幂等转订单 → 并发库存预占 → 履约计划 → 异常处理 → 应收与审计。**

该场景覆盖交易、供应链、状态机、并发、事务、事件、权限、读模型和前端操作流，同时避免落入“普通商城”的同质化实现。

## 3. 产品抽象

演示产品命名为 **CellarBridge（酒桥）**，其产品定位为：

> 面向酒饮进口商、供应链运营商及其商业客户的可解释 B2B 贸易协同平台。

平台不处理真实报关、真实支付或真实承运商接口，而是对这些外部能力建立清晰的端口、状态和模拟适配器。这样既保持业务完整，也不虚构监管能力。

## 4. 核心设计判断

- **模块化单体优先。** 当前规模和展示目标不支持为了形式拆分微服务；通过 Spring Modulith、模块 API、事件和独立数据归属保留拆分能力。
- **报价和订单必须保存快照。** 商品、成本、政策和客户资料会变化，历史商业承诺不能被后续数据覆盖。
- **贸易路径决策必须可解释。** 每个候选方案给出硬约束结果、评分分解、策略版本和人工覆盖原因。
- **库存正确性落在数据库。** 缓存只用于安全查询；预占使用确定性分配和原子条件更新。
- **异步流程采用至少一次语义。** 事件可能重复，消费端必须幂等；失败可恢复并可审计。
- **公开仓库只包含安全材料。** 原始截图、内部提示词和接力记录进入独立私有控制仓库，不依赖后期改写 Git 历史。

## 5. 研究限制

- 未获得公司内部组织结构、系统截图、数据模型、合同、业务规则或接口；
- 未验证所有品牌名称和公开数字在 2026 年是否仍然有效；
- 未将任何特定海关监管代码、税率、许可证规则写入核心业务逻辑；
- “贸易路径决策引擎”“客户信用规则”“异常 SLA”“应收模块”等是基于场景的演示增强，不宣称为研究对象的真实内部系统；
- 技术基线根据 2026-07-13 官方公开版本冻结，实施时只能在同一兼容范围内调整并记录原因。

## 6. 后续设计输入

本研究向后续文档提供以下冻结输入：

| 输入 | 决策 |
|---|---|
| 核心用户 | 销售、销售经理、贸易运营、仓库、财务、客户采购、管理员、审计 |
| 核心领域 | 报价、贸易路径、订单、库存预占、履约 |
| 支撑领域 | 客户、目录、异常、应收、报表 |
| 通用能力 | 身份、权限、审计、通知、可观测性 |
| 交付路径 | 上海一般贸易、宁波保税/跨境、香港自由贸易，以可配置模板表达 |
| 产品边界 | 不接真实海关、支付、ERP/WMS、承运商；提供模拟端口 |
| 数据边界 | 全部为合成数据 |
| 架构起点 | Java 模块化单体 + React SPA |
