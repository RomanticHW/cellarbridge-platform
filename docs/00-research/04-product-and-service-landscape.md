# 产品与服务图谱

## 1. 公开名称与可能职责

| 名称 | 公开描述 | 研究判断 | CellarBridge 对应能力 |
|---|---|---|---|
| FineWest / 优自西方 | 国际酒饮交易服务公司 | 企业与供应链服务主体 | 整体业务背景 |
| 大搜酒 WineMatcher | 精品酒饮供应链服务平台、内容驱动 B2B 交易 | 商品发现与 B2B 交易入口 | Catalog、Quotation、Customer Portal |
| ShunShip | 法国至上海零担国际物流 | 集运与国际物流服务 | Fulfillment 路径模板和里程碑 |
| 香港业务 | 酒饮自由贸易交付 | 区域交付与库存/履约路径 | HK_FREE_TRADE 路径 |
| 宁波跨境电商业务 | 宁波保税仓 B2B 交付 | 保税库存与交付路径 | NB_BONDED_B2B 路径 |
| 上海一般贸易交付 | 一般贸易交付 | 国内完税库存和国内履约 | SH_GENERAL_TRADE 路径 |

表中“研究判断”和“对应能力”属于产品抽象，不代表原系统模块名称。

## 2. 能力链而非页面列表

公开资料中的名称可以整理为六段能力链：

1. **供给发现：** 酒款、酒庄、产区、年份、包装、内容；
2. **商业匹配：** 客户需求、可售数量、客户专属报价；
3. **交易控制：** 毛利、折扣、审批、接受和订单；
4. **供给承诺：** 库存、在途、预售、海外代采；
5. **交付执行：** 仓库、集运、单证、物流节点和签收；
6. **经营闭环：** 异常、应收、审计和经营分析。

CellarBridge 的模块边界按这条价值链划分，而不是按“商品管理、订单管理、系统管理”的通用后台菜单划分。

## 3. 供给类型

根据发展时间线，产品需要区分以下供给语义：

| 供给类型 | 含义 | 是否进入 MVP |
|---|---|---|
| `DOMESTIC_ON_HAND` | 国内仓可直接预占的现货 | 是 |
| `BONDED_ON_HAND` | 保税或特定路径库存，需客户及路径资格 | 是 |
| `HONG_KONG_ON_HAND` | 香港库存或可在香港交付的供给 | 是，合成数据 |
| `IN_TRANSIT_PRESALE` | 已采购在途、按预计时间承诺 | 只读展示，P2 承诺逻辑 |
| `OVERSEAS_SOURCING` | 尚未形成库存，需要采购确认 | P2，不进入库存预占 |

MVP 只对前三类执行强库存预占。后两类可出现在目录和报价中，但订单确认前需要人工采购确认，避免把未来供给错误地当成现货。

## 4. 目录内容模型

WineMatcher 的“内容驱动”提示目录不应只有 SKU 名称和价格。产品目录至少表达：

- 酒庄/生产者；
- 国家、产区、子产区或法定产区文本；
- 酒款名称和类型；
- 年份；
- 容量、瓶数/箱、包装；
- 葡萄品种标签；
- 酒精度、评分或品鉴文本等可选内容；
- 供给类型、仓库、预计可用时间；
- 面向内部角色的成本和面向客户的报价，不在目录中混用。

项目不复制真实酒评内容，演示数据使用原创的简短描述。

## 5. 产品界面形态

### 5.1 内部运营控制台

面向销售、运营、仓库、财务和管理者，承载绝大部分功能。信息密度高，强调工作队列、状态、异常和责任人。

### 5.2 客户轻门户

面向商业客户采购人员，MVP 只提供：

- 查看已发送报价；
- 接受或拒绝报价；
- 查看订单和履约时间线；
- 查看应收状态，不执行真实支付。

客户门户与内部控制台共享 React 工程和设计系统，但路由、权限和可见字段严格分离。

## 6. 不选择的产品方向

- **消费者商城：** 无法体现多路径 B2B 供应链，且同质化严重；
- **酒款推荐算法：** 公开证据不足，且会把核心转向数据/模型而非 Java 企业工程；
- **完整 WMS：** 过度深入仓内作业，偏离交易协同；
- **完整 TMS：** 需要真实承运商与路径数据，演示成本高；
- **报关系统：** 涉及监管专业性和真实资质，不适合模拟成“可用产品”；
- **通用低代码审批：** 会稀释报价与供应链领域边界。
