# 产品路线图

## 1. 交付策略

以纵向业务切片交付，不采用“先建完所有 Entity/Repository/Controller，再做页面”的水平分层方式。每个切片至少包含产品行为、后端领域、API、持久化、React 交互、测试和文档。

## 2. 里程碑

### M0 — Design Baseline v1.0

交付：研究、产品、领域、架构、数据、OpenAPI/AsyncAPI、质量标准和实施任务包。无业务代码。

退出标准：文档链接和契约可校验；边界无已知矛盾；公开/私有资料分离。

### M1 — Executable Foundation

交付：

- Maven/Java 和 pnpm/React 工作区；
- PostgreSQL、Keycloak、Docker Compose core profile；
- 模块骨架、错误响应、租户上下文；
- OpenAPI 类型生成；
- CI、架构测试、Testcontainers 基线；
- 登录后空壳导航和健康页。

退出标准：一条命令启动；认证和双租户隔离测试通过；无业务 CRUD 冒充完成。

### M2 — Partner and Catalog Slice

交付：客户草稿—审核—激活；商品/SKU/供给搜索；角色权限和审计。

退出标准：销售和经理完成准入；搜索具有合成数据和查询性能证据。

### M3 — Quotation and Trade Planning Slice

交付：报价编辑、金额快照、路径硬约束/评分、审批和客户预览。

退出标准：路径结果确定、可解释；折扣/毛利/覆盖审批测试齐全。

### M4 — Acceptance, Order and Inventory Slice

交付：客户接受、幂等订单、可靠事件、原子全量预占、失败缺口。

退出标准：重复事件/并发接受只一个订单；热点库存测试无超卖。

### M5 — Fulfillment and Exception Slice

交付：路径模板、步骤依赖、SLA、公开里程碑、异常去重和恢复。

退出标准：主订单可完成；模拟失败能恢复；重启后事件不丢。

### M6 — Settlement and Reporting Slice

交付：应收、部分付款、冲正、驾驶舱读模型、统一时间线。

退出标准：订单到收款闭环；指标可追踪；字段权限测试通过。

### M7 — Engineering Evidence Release

交付：完整/核心 profile、可观测性、性能报告、安全扫描、SBOM、Playwright 演示、README 双语更新和 `v1.0.0` Release。

退出标准：新审阅者在 30 分钟内可启动并完成主路径；所有公开声明有证据。

## 3. 延后能力

| 能力 | 原因 | 进入条件 |
|---|---|---|
| 部分预占/拆单 | 会显著扩大订单和履约状态空间 | P1 全量预占稳定后另建 ADR |
| 真实 WMS/TMS/海关接口 | 无公开真实契约且易虚构 | 获得合法沙箱契约后 |
| Elasticsearch/OpenSearch | 目录规模不要求独立集群 | PostgreSQL 查询/运维证据不足 |
| 微服务拆分 | 当前团队和演示规模无收益 | 模块负载/发布/团队独立性证据 |
| 工作流引擎 | P1 状态机足够，避免黑盒 | 流程动态配置需求明确 |
| 真实支付/财务总账 | 安全与合规范围过大 | 独立产品目标和测试环境 |
| AI 推荐/自动定价 | 与核心正确性无关 | 有可解释数据集和产品价值 |
| 移动端原生应用 | 运营场景桌面优先 | 客户门户移动需求验证 |

## 4. 技术展示优先级

1. 可解释业务规则；
2. 聚合不变量和状态机；
3. 幂等与数据库最终防线；
4. 库存并发正确性；
5. 可靠事件和失败恢复；
6. 模块架构自动验证；
7. 租户/字段安全；
8. React 完整业务闭环；
9. 可观测性和性能证据；
10. 中间件数量。

## 5. 发布纪律

- 每个里程碑发布一个可运行 tag；
- README 能力状态只由实现清单生成或人工双重检查；
- 未实现能力写 `Designed`/`Planned`，不写 `Available`；
- 公开仓库不提交内部 Prompt、原始研究截图、个人接力记录和临时密钥；
- 任何影响核心边界的变更必须先更新 ADR 与 Design Baseline 版本。
