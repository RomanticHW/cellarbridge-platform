# 产品范围与原则

## 1. 范围分层

### P0 — 设计与工程基线

- 研究、产品、领域、架构和契约；
- 仓库治理、质量门禁和交付计划；
- Java/React 工程骨架、身份、数据库和可观测性基础；
- 一键启动和文档验证。

### P1 — 可演示核心闭环

- OIDC 登录、租户和角色；
- 商业客户创建、审核、激活、停用和路径资格；
- 酒款、SKU、仓库、供给池和库存批次；
- 目录检索和可承诺数量查询；
- 报价草稿、版本、路径评估、审批、发送和客户接受；
- 幂等报价转订单；
- 订单全量库存预占；
- 路径化履约计划、步骤和公开里程碑；
- 异常分派、处理、恢复和关闭；
- 应收、部分付款、全额付款和冲正；
- 审计时间线和经营看板；
- 主路径 Playwright 测试与合成数据。

### P2 — 工程强化

- 在途预售供给展示和采购确认；
- Kafka 完整配置与外部事件消费者；
- Redis 安全读缓存；
- PostgreSQL 全文/模糊搜索优化；
- SLA 调度与批量异常生成；
- 性能压测、故障注入、PIT 变异测试；
- OpenTelemetry 全链路、Grafana 仪表盘；
- SBOM、容器扫描、依赖更新策略；
- 面向评审的截图、视频或 Playwright trace。

### P3 — 可选扩展

- 多仓分配和拆单；
- 在途预售承诺与采购订单；
- 退货与索赔；
- 多币种结算和汇兑损益；
- 外部 ERP/WMS/TMS 适配器；
- 高级定价和客户合同价；
- 服务提取演示，不作为默认架构。

## 2. 明确排除

| 排除项 | 原因 |
|---|---|
| 真实酒类商品交易 | 法律、资质和数据风险，不是技术演示目标 |
| 真实支付网关 | 增加凭据与合规风险；付款记录足以完成闭环 |
| 真实海关申报 | 需要专业规则和授权，不应以模拟结果冒充可用系统 |
| 真实税务发票和总账 | 会把范围扩展到财务系统 |
| 完整采购、WMS、TMS | 通过端口协作，避免三个项目合并成一个 |
| 消费者商城和营销 | 与核心 B2B 协同价值不一致 |
| 推荐模型和自动定价 | 证据不足且偏离 Java 企业工程主线 |
| 区块链溯源 | 没有明确业务必要性 |
| 默认微服务部署 | 运行成本高且边界收益不足 |

## 4. 关键产品规则

1. 只有 `ACTIVE` 客户可以收到新报价；
2. 客户必须具备所选交付路径资格；
3. 报价至少包含一行，数量和金额必须为正；
4. 同一报价使用单一报价币种；
5. 报价发送前必须通过所有需要的审批；
6. 客户只能接受 `SENT` 且未过期的报价；
7. 接受后的报价不可修改；
8. 同一报价最多转换为一个订单；
9. 订单商业条件来自报价快照，不在转换时重新定价；
10. P1 只支持订单行全部预占成功或全部失败；
11. 预占数量不得超过批次可用量；
12. 履约步骤只能在依赖步骤完成后开始；
13. 订单完成要求履约必需步骤完成；
14. 付款总额不得静默超过应收；
15. 财务和审计记录不得物理删除；
16. 所有写操作在租户上下文中执行；
17. 人工覆盖、豁免、取消和冲正必须记录原因。

## 5. 简化决定

为保证个人项目可交付，P1 做出以下简化：

- 单个订单只能选择一个贸易路径；
- 单个订单使用一种币种；
- 只自动预占现货供给；
- 不支持部分预占和自动拆单；
- 报价审批为规则触发的单级经理审批；
- 客户信用为配置值，不连接外部征信；
- 履约外部结果由可控模拟器产生；
- 付款由财务手工登记；
- 只实现一个租户的演示数据，但代码和测试保持多租户边界；
- 页面默认中文，技术术语和 API 使用英文稳定标识。

## 6. 产品设计原则

### 6.1 不把未来状态当成当前事实

“目录可见”“预计可用”“已报价”“已预占”“已出库”必须是不同状态和来源。

### 6.2 自动化不取消人工责任

系统可以推荐路径和触发审批，但人工覆盖必须有理由，且不能覆盖硬约束失败。

### 6.3 失败是正常业务分支

库存不足、路径不合格、外部步骤失败和客户拒绝不是异常堆栈，而是明确的业务结果。只有不可预期技术失败进入技术告警。

### 6.4 交易历史不可被主数据重写

客户名称、地址、SKU 描述、价格构成、策略版本和路径信息在报价/订单中保存快照。

### 6.5 公开状态与内部状态分离

客户只看到适合公开的里程碑，不看到成本、内部审批、仓库细节或异常根因。

### 6.6 每项复杂度都要有评审证据

引入 Kafka、Redis、调度器、缓存和异步投影前，必须指定业务问题、失败模式、测试和可观察指标。
