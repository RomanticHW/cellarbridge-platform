# 非功能需求规格

## 1. 目标与适用范围

本文件定义 CellarBridge 可验证的质量属性。目标是形成可信的作品级企业系统，而不是虚构互联网规模。容量目标用于本地和 CI 验证，不能被描述为生产承诺。

## 2. 可用性与恢复

| ID | 要求 | 目标/验证 |
|---|---|---|
| NFR-AVL-001 | 核心 API 应暴露 liveness/readiness | Docker Compose 和 CI 可检查 |
| NFR-AVL-002 | 数据库暂时不可用时应用不得报告 ready | 停库故障测试 |
| NFR-AVL-003 | 异步处理失败应重试并进入可观察失败状态 | 重试次数、最后错误、下一次时间可查 |
| NFR-AVL-004 | 重启后未发布事件和未完成工作可恢复 | 进程终止后恢复测试 |
| NFR-AVL-005 | 演示环境目标 RPO 为数据库最近成功备份，RTO 不作生产级承诺 | 提供备份/恢复说明 |

## 3. 性能和容量

设计数据集：

- 1 个演示租户，另含 1 个隔离测试租户；
- 5,000 个商品、15,000 个 SKU；
- 20 个仓库/供给池、100,000 个库存批次；
- 100,000 份报价、60,000 个订单、1,000,000 条审计/事件记录；
- 本地压测并发用户 50，库存热点并发命令 100~500。

| ID | 要求 | P95 目标（演示硬件） |
|---|---|---|
| NFR-PERF-001 | 商品搜索 | < 500 ms |
| NFR-PERF-002 | 报价详情读取 | < 300 ms |
| NFR-PERF-003 | 路径评估（3~10 条路径、20 行以内） | < 500 ms |
| NFR-PERF-004 | 普通命令 API（不含外部异步完成） | < 800 ms |
| NFR-PERF-005 | 驾驶舱读模型 | < 1 s |
| NFR-PERF-006 | 并发库存测试 | 无超卖、无死锁未处理、结果可解释 |

延迟目标不包括首次容器拉取、JVM 冷启动和开发机器资源争用。测试报告必须记录硬件、数据规模和命令。

## 4. 一致性与正确性

| ID | 要求 |
|---|---|
| NFR-COR-001 | 一个已接受报价至多创建一个订单 |
| NFR-COR-002 | 库存预占后 `0 <= reserved <= on_hand` 永远成立 |
| NFR-COR-003 | 订单和接受报价的商业快照不可被主数据更新改变 |
| NFR-COR-004 | 外部消息采用至少一次投递假设，消费者必须幂等 |
| NFR-COR-005 | 调度任务可重复执行且不产生重复终态副作用 |
| NFR-COR-006 | 金额运算具有固定币种精度、舍入和属性测试 |
| NFR-COR-007 | 所有业务时间使用 UTC 持久化，界面按用户时区显示 |

## 5. 安全

| ID | 要求 | 验证 |
|---|---|---|
| NFR-SEC-001 | 所有非公开 API 默认需要认证 | 安全测试 |
| NFR-SEC-002 | 每次读取和写入均强制租户隔离 | 双租户集成测试 |
| NFR-SEC-003 | 权限码与字段可见性由后端执行 | 越权/字段泄露测试 |
| NFR-SEC-004 | 客户门户令牌/链接不可猜测、可过期、可撤销 | 负向测试 |
| NFR-SEC-005 | 密钥只由环境/秘密存储注入，不进入仓库 | secret scan |
| NFR-SEC-006 | 日志不得包含访问令牌、密码、完整个人信息、成本和毛利 | 日志测试/评审 |
| NFR-SEC-007 | 依赖、容器和 SBOM 进入 CI 检查 | GitHub Actions 证据 |
| NFR-SEC-008 | 所有输入有长度、格式、枚举和业务校验 | API 负向测试 |
| NFR-SEC-009 | 管理/重放/演示重置等高危操作需专门权限和审计 | 权限矩阵测试 |

## 6. 隐私与数据治理

- 公开仓库只包含合成数据；
- 个人信息仅保留业务必要字段；
- API 返回最小字段，读模型按角色投影；
- 删除请求在演示系统中以匿名化/停用策略说明，不声称满足特定司法辖区所有法规；
- 审计记录保留业务 ID 和动作摘要，敏感值采用掩码或差异摘要；
- 原始研究截图、内部 Prompt 和接力记录不得进入公开仓库。

## 7. 可维护性

| ID | 要求 |
|---|---|
| NFR-MNT-001 | 模块边界由 Spring Modulith 和 ArchUnit 自动验证 |
| NFR-MNT-002 | 公开模块 API 与 internal 实现隔离 |
| NFR-MNT-003 | 数据库迁移合并后不可改写 |
| NFR-MNT-004 | API/event 契约通过自动校验和兼容性检查 |
| NFR-MNT-005 | 关键业务规则具有命名测试和需求 ID |
| NFR-MNT-006 | 关键架构变化先更新 ADR |
| NFR-MNT-007 | README 的实现状态由发布检查验证，禁止夸大 |

质量目标（初始）：

- 领域核心模块行覆盖率 ≥ 85%，分支覆盖率 ≥ 75%；
- 全仓库覆盖率仅作辅助，不通过低价值测试追求百分比；
- mutation testing 在报价、路径策略、订单幂等和库存算法上分阶段启用；
- 静态分析高严重度问题为零。

## 8. 可观测性

- 每个入站请求具有 trace ID；
- 关键命令具有 correlation ID 和 causation ID；
- 结构化日志包含租户匿名标识、模块、用例、结果和错误码，不包含敏感业务值；
- 指标至少覆盖 API 延迟/错误、数据库池、事件积压、重试、报价转换、库存预占、履约逾期和开放异常；
- 主链路可在追踪系统中看到 API → 本地事务 → 事件发布 → 消费者；
- 告警规则作为演示配置，明确不代表生产阈值。

## 9. 可访问性与易用性

| ID | 要求 |
|---|---|
| NFR-UX-001 | 主演示页面目标满足 WCAG 2.1 AA 的关键要求 |
| NFR-UX-002 | 所有核心操作可用键盘完成 |
| NFR-UX-003 | 表单错误与字段关联且不只依赖颜色 |
| NFR-UX-004 | 页面具有加载、空、失败、无权限和并发冲突状态 |
| NFR-UX-005 | 重要状态、金额、日期和业务编号具有统一格式 |
| NFR-UX-006 | 客户可见文案不暴露内部技术或策略字段 |

## 10. 可移植性与本地体验

- 支持 Linux/macOS；Windows 通过 Docker Desktop + WSL2 验证；
- 核心 profile 目标一条命令启动 PostgreSQL、Keycloak、后端和前端；
- 完整 profile 增加 Kafka、Redis 和可观测性组件；
- Java、Node 和容器版本固定；
- 提供 `make doctor`、`make up-core`、`make verify`、`make demo-reset` 等统一入口；
- 不依赖任何付费 SaaS 或私有云资源完成主演示。

## 11. 兼容性

- OpenAPI 采用向后兼容扩展优先；
- 破坏性 API 变化需版本决策；
- 事件 schema 不删除/重解释已有字段；
- 数据库迁移支持前后版本短暂滚动兼容作为设计目标，但模块化单体演示不伪装已完成生产零停机；
- 浏览器支持最近两个稳定版本的 Chrome、Edge 和 Firefox。

## 12. 质量属性权衡

优先级：

1. 业务正确性和安全；
2. 可解释性和可审阅性；
3. 可维护性和本地运行；
4. 性能；
5. 横向扩展；
6. 功能广度。

因此，项目宁可明确拒绝库存不足，也不以最终一致的“暂时成功”隐藏超卖；宁可使用模块化单体保持可运行性，也不提前支付微服务运维成本。
