# 产品愿景

## 1. 产品名称

- 英文：CellarBridge
- 中文：酒桥
- 描述：面向酒饮进口与供应链运营的 B2B 贸易协同平台
- 核心口号：从询报价到交付，每个承诺都有依据，每个状态都可追踪

## 2. 问题陈述

专业酒饮 B2B 交易通常不是“选商品、付款、发货”的简单流程。一个客户需求可能同时涉及：

- 多年份、多包装和多供给位置；
- 国内现货、保税库存、香港库存、在途或海外代采；
- 客户专属价格、折扣、毛利和付款条款；
- 不同交付路径的资格、成本、时效和操作复杂度；
- 销售、经理、贸易运营、仓库、财务和客户之间的协同；
- 报价接受后不能重复下单、不能超卖、不能丢失履约状态；
- 异常必须有人负责、可以恢复，并留下完整审计证据。

若这些信息分散在聊天、表格、邮件和独立系统中，容易出现报价口径不一致、重复订单、库存承诺失真、履约状态不透明和责任追踪困难。

## 3. 产品愿景

> CellarBridge 将客户、酒款、货源、报价、贸易路径、订单、库存、履约和应收组织成一条可解释、可验证的业务链路，使销售承诺可以被运营执行，使履约事实可以被客户和管理者理解。

## 4. 目标用户

### 内部角色

- 销售代表：快速响应客户需求并形成准确报价；
- 销售经理：控制折扣、毛利、特殊条款和路径覆盖；
- 贸易运营：选择并推进交付路径，处理外部依赖；
- 仓库操作员：确认批次、预占结果、出库和异常；
- 财务人员：维护应收、付款和逾期状态；
- 租户管理员：配置用户、角色、基础政策和演示数据；
- 审计/只读角色：查看状态、决策依据和操作证据。

### 外部角色

- 商业客户采购人员：查看并接受报价、跟踪订单和履约状态。

## 5. 价值主张

### 5.1 对销售

- 从统一目录和可承诺供给开始报价；
- 自动获得可用交付路径及拒绝原因；
- 明确哪些条件触发审批；
- 报价接受后无需重复录入订单；
- 对客户可说明价格与交付承诺的依据。

### 5.2 对运营

- 订单和库存状态由系统事件衔接；
- 不同路径自动生成对应履约步骤；
- 所有中间状态、外部调用和失败可见；
- 逾期和失败自动进入异常工作队列；
- 人工覆盖和恢复操作保留原因。

### 5.3 对管理者

- 看到报价审批周期、路径覆盖、预占成功率和履约 SLA；
- 可追踪低毛利、高折扣和特殊条款；
- 不需要跨表手工拼接订单时间线；
- 通过架构和审计约束降低隐性风险。

### 5.4 对客户

- 获得清晰、有效期明确的商业报价；
- 能看到选定交付方案和预计时间；
- 接受操作可追踪，不因重复点击产生重复订单；
- 订单和异常进展透明。

## 6. 产品目标

| ID | 目标 | 可验证结果 |
|---|---|---|
| PG-01 | 建立从报价到交付的完整业务闭环 | 演示脚本可由新用户独立完成 |
| PG-02 | 保证关键交易正确性 | 幂等、并发、状态机和事务测试通过 |
| PG-03 | 让贸易路径决策可解释 | 每个候选方案有规则结果、评分和版本 |
| PG-04 | 将长流程失败转化为可管理工作 | 失败进入异常中心并支持恢复 |
| PG-05 | 提供可审阅的企业级工程证据 | 文档、契约、测试、指标和代码一致 |
| PG-06 | 保持本地运行门槛可控 | 核心配置一条命令启动，完整配置可选 |

## 7. 非目标

CellarBridge 不以以下事项为目标：

- 成为真实酒类销售平台；
- 模拟完整海关、税务或许可证法规；
- 替代 ERP、WMS、TMS 或财务总账；
- 实现消费者营销、购物车和促销；
- 建立机器学习推荐或自动定价；
- 用微服务数量证明架构能力；
- 为了技术栈丰富而引入没有业务理由的中间件。

## 8. 北极星结果与辅助指标

北极星结果不是交易额，而是：

> **一份被客户接受的报价能够以可证明正确的方式转换为一个订单，并沿选定路径完成库存承诺和履约。**

辅助指标：

- 报价草稿到发送的中位时间；
- 审批等待时间；
- 贸易路径硬约束拒绝分布；
- 人工覆盖推荐比例；
- 重复转换请求的幂等命中率；
- 库存预占成功率；
- 履约步骤按时完成率；
- 开放异常和平均解决时长；
- 应收逾期比例；
- 审计事件缺失数，目标为零。

这些指标只用于演示系统的业务可观测性，不对应研究对象的真实 KPI。

## 9. 产品原则

1. **承诺必须可兑现。** 搜索结果、报价数量和最终预占是不同层级的事实。
2. **决定必须可解释。** 自动推荐不能只返回一个黑盒结果。
3. **历史必须保持原意。** 交易快照不随主数据变化。
4. **正确性优先于瞬时成功。** 库存不足应明确失败，而不是延迟暴露超卖。
5. **人工操作必须有责任。** 覆盖、豁免、取消和冲正必须记录原因与操作者。
6. **状态必须可恢复。** 长流程失败不能依赖修改数据库解决。
7. **权限必须落在后端。** UI 隐藏不是安全控制。
8. **复杂度必须有业务回报。** 每项基础设施都对应明确问题和验收证据。
9. **公开内容必须可信。** 只展示已完成能力，全部数据为合成数据。
