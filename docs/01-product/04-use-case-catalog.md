# 用例目录

## 1. 用例建模规则

本目录使用 `UC-<领域>-<序号>` 标识用例。每个用例必须能够追溯到能力、功能需求、接口契约、权限和验收场景。用例描述业务目标，而不是页面按钮或 Controller 方法。

统一字段：

- **主要参与者**：发起并获得业务价值的角色；
- **前置条件**：系统在开始用例前必须满足的事实；
- **触发**：用例开始的业务事件；
- **主成功场景**：不省略关键校验和状态变化；
- **替代/失败场景**：可预期的业务分支；
- **后置条件**：成功或失败后系统必须保持的事实；
- **审计要求**：必须保留的证据；
- **幂等要求**：重复请求是否允许及如何判断重复；

## 2. 身份与客户

### UC-IAM-001 登录并建立租户上下文

- **主要参与者**：任意平台用户。
- **前置条件**：用户存在于身份提供方；其组织在 CellarBridge 中处于启用状态。
- **触发**：用户打开受保护页面。
- **主成功场景**：
  1. 浏览器通过 OIDC Authorization Code + PKCE 登录；
  2. 前端取得访问令牌；
  3. 后端验证签名、受众、签发者和时效；
  4. 系统从声明和本地映射建立 `TenantContext`、用户身份和权限集合；
  5. 返回当前用户可访问的导航与工作队列摘要。
- **失败场景**：租户停用、用户未映射、令牌过期、权限不足。
- **后置条件**：失败不得创建本地会话状态或泄露其他租户数据。
- **审计要求**：记录安全相关失败的摘要，不记录令牌。

### UC-PAR-001 创建商业客户申请

- **主要参与者**：销售代表。
- **前置条件**：销售代表拥有 `partner:create` 权限。
- **主成功场景**：
  1. 输入客户法定名称、简称、类型、联系人、地址、默认币种和拟申请付款条款；
  2. 系统在租户内检查重复法定名称和统一识别号；
  3. 创建 `Partner` 草稿；
  4. 销售补充渠道资格与贸易路径申请；
  5. 提交审核，状态变为 `PENDING_REVIEW`；
  6. 创建审核工作项并通知销售经理。
- **替代场景**：发现潜在重复时允许保存草稿，但不能提交，必须选择现有客户或提供差异说明。
- **审计要求**：保存创建者、提交者、字段变更和重复检查结果。

### UC-PAR-002 审核并激活客户

- **主要参与者**：销售经理或租户管理员。
- **前置条件**：客户处于 `PENDING_REVIEW`。
- **主成功场景**：
  1. 审核基本资料、付款条款、信用额度和路径资格；
  2. 可批准、退回修改或拒绝；
  3. 批准后客户状态为 `ACTIVE`，生成不可变审核记录；
  4. 发布 `PartnerActivatedV1`。
- **规则**：审核人不能是当前申请的提交人；演示配置可为小团队启用“双人原则豁免”，但必须留下原因。

## 3. 商品、供给与查询

### UC-CAT-001 维护酒款与 SKU

- **主要参与者**：商品运营。
- **前置条件**：拥有目录维护权限。
- **主成功场景**：创建酒款、产区、生产者、分类和内容信息；在其下创建以年份、容量、包装数量和包装形态区分的 SKU；激活后进入可检索目录。
- **规则**：同一租户内 SKU 编码唯一；年份可为 `NV`；容量和箱规必须为正数；已被交易快照引用的 SKU 不物理删除。

### UC-INV-001 查询可承诺供给

- **主要参与者**：销售代表。
- **触发**：在报价编辑器搜索 SKU。
- **主成功场景**：
  1. 按关键字、年份、产区、生产者、价格区间和供给类型检索；
  2. 返回目录主数据、供给位置、显示级别的可用量、预计到货日期和更新时间；
  3. 明确区分 `即时可预占`、`需人工确认` 和 `仅参考`；
  4. 销售将目标数量加入报价草稿。
- **规则**：此查询不是库存承诺；页面必须展示“以订单预占结果为准”。

## 4. 报价与贸易路径

### UC-QUO-001 创建客户报价

- **主要参与者**：销售代表。
- **前置条件**：客户为 `ACTIVE`，至少具备一种可用路径资格。
- **主成功场景**：
  1. 选择客户、币种和期望交付日期；
  2. 添加一个或多个 SKU、数量和可选供给偏好；
  3. 价格服务根据价目、客户等级和人工折扣形成行项目价格快照；
  4. 贸易规划引擎评估候选路径；
  5. 销售选择一个有效路径，或使用推荐路径；
  6. 系统计算小计、费用、折扣、税费占位项和总额；
  7. 保存草稿。
- **规则**：金额使用币种最小单位或 `BigDecimal` 且统一舍入；每一行必须保留定价输入和版本。

### UC-TRD-001 评估贸易路径

- **主要参与者**：报价应用服务（系统参与者）。
- **输入**：客户资格、目的地、SKU 与供给、期望日期、付款条款、策略版本。
- **主成功场景**：
  1. 对 `SH_GENERAL_TRADE`、`NB_BONDED_B2B`、`HK_FREE_TRADE` 等已配置路径执行硬约束；
  2. 为不满足的路径输出稳定拒绝码和解释；
  3. 对有效路径按成本、时效、供给置信度和操作复杂度评分；
  4. 返回推荐路径、完整候选列表、策略版本和输入摘要；
  5. 将结果作为报价快照的一部分保存。
- **规则**：不在代码中模拟真实海关法规、税率或许可证结论；规则来自演示配置。

### UC-QUO-002 提交报价审批

- **主要参与者**：销售代表。
- **前置条件**：报价草稿完整且至少一个有效路径。
- **主成功场景**：
  1. 系统执行完整校验；
  2. 审批策略根据折扣、预计毛利、付款条款、人工路径覆盖和异常价格决定是否审批；
  3. 无需审批则直接成为 `APPROVED`；
  4. 需要审批则成为 `PENDING_APPROVAL`，生成一个或多个审批项；
  5. 发布相应事件并通知审批人。
- **失败场景**：无有效路径、客户状态变化、报价行价格过期、期望日期不满足。

### UC-QUO-003 审批报价

- **主要参与者**：销售经理。
- **前置条件**：存在分配给当前角色的待审批项。
- **主成功场景**：查看触发原因、毛利/折扣摘要、路径解释和历史；批准、退回修改或拒绝；决定和意见写入不可变审批记录。
- **规则**：审批动作必须带乐观版本；重复动作返回当前最终结果，不创建第二条相互矛盾的决定。

### UC-QUO-004 发送并由客户接受报价

- **主要参与者**：销售代表、客户采购人员。
- **主成功场景**：
  1. 销售发送已批准报价；
  2. 系统生成不可猜测的客户访问上下文，并显示有效期、条款和公开路径信息；
  3. 客户接受或拒绝；
  4. 接受时再次检查报价未过期、未撤销、版本匹配；
  5. 状态变为 `ACCEPTED` 并发布事件。
- **幂等要求**：同一接受命令重复提交返回同一接受结果；不得重复触发后续订单。

## 5. 订单与库存

### UC-ORD-001 将接受报价转换为订单

- **主要参与者**：系统，销售代表可查看结果。
- **触发**：`QuotationAcceptedV1` 或显式重试命令。
- **主成功场景**：
  1. 以报价 ID 作为业务幂等键检查是否已有订单；
  2. 读取报价发布的不可变商业快照；
  3. 创建唯一订单号和 `TradeOrder`；
  4. 保存客户、商品、价格、路径和条款快照；
  5. 在同一事务中记录可靠事件发布；
  6. 返回已有或新建订单。
- **最终保证**：一个报价至多对应一个订单；数据库唯一约束为最终防线。

### UC-INV-002 预占订单库存

- **主要参与者**：库存模块（系统参与者）。
- **前置条件**：订单处于 `PENDING_RESERVATION`；供给类型支持自动预占。
- **主成功场景**：
  1. 按仓库、到货日期、批次和稳定 ID 排序候选批次；
  2. 对每个订单行执行原子条件扣减/增加预占；
  3. 全部行成功后创建预占单并发布 `InventoryReservationConfirmedV1`；
  4. 任一行不足则回滚整个事务，发布失败事实并给出缺口。
- **并发规则**：任何时刻 `reserved_quantity <= on_hand_quantity`；不使用 Redis 锁保证正确性。
- **P1 范围**：全量预占或全量失败，不自动拆单和部分预占。

### UC-INV-003 释放或消费预占

- **主要参与者**：履约模块、仓库操作员。
- **规则**：取消/失败释放未消费预占；出库确认将预占转为已消费；动作幂等且不得产生负数。

## 6. 履约与异常

### UC-FUL-001 生成履约计划

- **主要参与者**：履约模块。
- **触发**：库存预占成功。
- **主成功场景**：按订单路径和模板版本生成步骤、依赖、负责人角色、计划完成时间、公开/内部可见性；状态为 `READY`；发布计划创建事件。

### UC-FUL-002 推进履约步骤

- **主要参与者**：贸易运营、仓库操作员、系统模拟适配器。
- **主成功场景**：领取工作项，开始、完成或失败；系统验证依赖和权限；完成后解锁后继步骤；所有公开里程碑进入客户时间线。
- **规则**：跳过、回滚和人工完成必须由模板允许并记录原因。

### UC-EXC-001 创建并处理异常

- **主要参与者**：系统、贸易运营。
- **触发**：步骤失败、SLA 超时、库存短缺、事件处理失败达到阈值。
- **主成功场景**：创建去重异常；分派、确认、调查、选择恢复动作；恢复成功后关闭并关联原业务对象。
- **规则**：关闭异常不等于自动修复业务状态；恢复动作必须显式执行和验证。

## 7. 应收、审计与报表

### UC-SET-001 创建应收并登记付款

- **主要参与者**：财务人员。
- **主成功场景**：订单达到配置节点后创建应收；登记部分或全额付款；金额达到应收后状态为 `PAID`；错误付款通过冲正记录纠正。
- **非目标**：不接真实支付通道，不生成会计总账或真实税务凭证。

### UC-AUD-001 查看统一业务时间线

- **主要参与者**：授权内部用户和审计人员。
- **主成功场景**：按订单/报价查询来自多个模块的事件投影；显示操作者、时间、行为、状态前后、关联 ID 和相关追踪 ID；敏感字段按权限脱敏。

### UC-REP-001 查看经营与履约驾驶舱

- **主要参与者**：销售经理、运营经理。
- **指标**：报价周期、审批积压、路径选择分布、转换幂等命中、预占成功率、开放异常、步骤逾期、应收状态。
- **规则**：指标来自专用读模型，不在请求时跨模块联表。
